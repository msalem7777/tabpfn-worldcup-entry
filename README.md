# TabPFN World Cup 2026 Prediction Pipeline

This repository contains the code behind my entry to the Prior Labs **TabPFN Football Prediction Competition**.

The project did not begin as a game-theoretic exercise. It began as a conventional football forecasting pipeline and developed one layer at a time as each previous version exposed a limitation:

1. build a reproducible TabPFN football baseline;
2. separate attack and defence using time-varying Dixon–Coles ratings;
3. learn persistent team-specific errors through chronological residual features;
4. add compact, time-aware tactical style categories;
5. test whether a structural reference model could improve calibration;
6. with only two matches left and a winner-take-all prize, separate the **probability forecast** from the **competition decision**.

The repository has two goals:

- **Reproducibility:** every submitted probability is generated by code from documented inputs.
- **Narrative:** the README records why each layer was added and what problem it was intended to solve.

The target throughout is the **90-minute regulation result**: home win, draw, or away win. Extra time and penalties do not change the scored class.

> **Repository policy:** generated model outputs, diagnostics, datasets, and submission CSVs are deliberately not committed. They are ignored by Git and regenerated locally from the code and public inputs.

# The modeling story

## 1. Baseline TabPFN football features

The starting point followed the competition’s supplied football setup and the public `martj42/international_results` match history. Each match was represented by information available before kickoff, including:

- Elo difference;
- recent form and goal difference;
- home and away win rates;
- head-to-head summaries;
- rest days;
- neutral venue;
- match importance.

This was a reasonable baseline, but most team strength was compressed into generic rolling summaries. It did not explicitly distinguish attack from defence, and it could not learn that the model repeatedly over- or under-estimated a particular team in a systematic way.

## 2. Time-varying Dixon–Coles ratings

The first extension was a time-decayed Dixon–Coles score model.

`01_dixon_coles_ratings.py` estimates:

- team attack strength;
- team defence strength;
- home advantage;
- the low-score dependence parameter `rho`.

`02_dixon_coles_rating_history.py` then refits those quantities at historical checkpoints. Training rows are joined to the latest rating available **strictly before the match**, using a backward as-of join.

This adds a structured football prior while avoiding future leakage. It also lets TabPFN use separate attack and defence information rather than relying only on a single overall rating.

## 3. Chronological residual-bias features

The next question was not merely “how strong is the team?” but:

> Where does the current model repeatedly make the same kind of mistake?

`03_predict_finals.py` creates chronological out-of-fold predictions. Each validation period is predicted by a model trained only on earlier matches. Those genuine out-of-sample errors are converted into expanding team-level residual summaries, including attack and defence residual tendencies.

Conceptually:

```text
v1 = baseline engineered football features
v2 = v1 + time-varying Dixon–Coles attack/defence ratings
v3 = v2 + chronological team residual-bias features
```

The residual layer is important because it lets the model learn persistent mispricing without using the current match outcome to construct its own feature.

## 4. Time-aware style keywords

Historical results do not fully describe *how* a team creates those results. The next layer added compact tactical descriptors as categorical TabPFN inputs.

The style data are generated by:

```powershell
python 02b_build_time_aware_style_keywords.py
```

The script defines dated tactical eras for the 2026 knockout-stage teams and joins them to historical match rows using the match date. This replaced an earlier leakage-prone idea in which one modern team description could be copied backward across the full historical record.

For each side of each match, the augmentation adds:

- three controlled-vocabulary style keywords;
- a canonical keyword triplet;
- the active tactical era label;
- the era start and end dates;
- a short style summary;
- confidence and provenance fields;
- an explicit timing-status field.

The controlled vocabulary is deliberately small so that TabPFN receives stable categorical levels rather than a sparse collection of nearly unique phrases. Example categories include:

```text
compact, counter, creative, direct, disciplined, efficient,
flexible, physical, possession, press, structured, tempo,
transition, vertical, wide, workrate
```

The generated local files are:

```text
data/raw/results_with_style_keywords_time_aware.csv
data/interim/team_style_keyword_eras_round32_time_aware.csv
```

Both are generated artifacts and remain outside version control.

The join is time-aware:

- a match receives only the style era active on that date;
- rows before the first supported era remain unlabelled;
- teams outside the modeled field receive an explicit `not_round32_team` status;
- missing historical coverage is not replaced by a modern tactical label.

For the final matchup, the compact vocabulary captured broad contrasts such as:

- **Spain:** possession, press, vertical;
- **Argentina:** creative, possession, structured.

The style layer was not intended to replace performance data. It gave TabPFN a compact categorical signal about matchup structure that ordinary ratings and rolling form could not express directly.

## 5. Structural calibration

The next experiment asked whether the neural probabilities should be pulled toward a simpler structural model when the two disagreed strongly.

`04_anchor_gated_calibration_finals.py` freezes the TabPFN forecast and fits a gated geometric pool between:

```text
TabPFN probability vector  <---- learned gate ---->  structural reference vector
```

The reference can be Dixon–Coles or ordered logit. The gate uses interpretable disagreement and uncertainty signals such as:

- absolute Elo difference;
- KL divergence between TabPFN and the reference;
- excess favourite uncertainty;
- selected form and head-to-head variables.

Calibration is evaluated on chronological out-of-fold predictions and selected by realized-outcome log loss. This prevents the calibration layer from merely forcing agreement with a preferred narrative.

Even when the raw v3 model remained the predictive anchor, this stage provided a disciplined test of whether apparent underconfidence was actually a calibration defect.

# 6. From forecasting to a winner-take-all decision

## Leaderboard evolution

The strategic layer was motivated by the way the competition evolved, not by a pre-planned desire to submit extreme probabilities.

The displayed cumulative mean log losses and ranks were:

| Checkpoint | Matches scored | Our entry | Rank | Reference leader | Rank |
|---|---:|---:|---:|---:|---:|
| After Round of 32 | 16 | 0.841 | 8th | 0.812 | 4th |
| After Round of 16 | 24 | 0.808 | 2nd | 0.802 | 1st |
| After quarterfinals | 28 | 0.829 | 3rd | 0.824 | 2nd |
| After semifinals | 30 | 0.839 | 3rd | 0.824 | 1st |

A displayed mean-log-loss difference is not itself the amount that must be recovered. The leaderboard averages across all scored matches. At each checkpoint, the implied cumulative total-log-loss deficit was reconstructed as

```text
total deficit = (our displayed mean - reference leader's displayed mean) × matches scored
```

which gives:

| Checkpoint | Mean gap | Implied total-log-loss deficit |
|---|---:|---:|
| After Round of 32 | 0.029 | 0.464 |
| After Round of 16 | 0.006 | 0.144 |
| After quarterfinals | 0.005 | 0.140 |
| After semifinals | 0.015 | 0.450 |

The progression matters:

- after the Round of 32, the gap was substantial;
- by the Round of 16, the pipeline had nearly caught the leader;
- the quarterfinal gap remained almost unchanged;
- the semifinal round widened the deficit back to approximately **0.45 total log-loss units**;
- only the third-place match and final remained.

At that point, submitting the raw model probabilities would still be optimal for expected predictive score, but it could be strategically inadequate. If our probabilities were similar to the reference leader's, even a correct forecast might not create enough relative gain to erase the existing deficit.

## Why expected log loss was no longer the full objective

For an ordinary forecast, the proper action is the probability vector believed to describe the match:

```text
minimize expected log loss
```

But the competition prize was awarded only to first place. There was no prize distinction between second and lower positions. With one decision left, the relevant utility became approximately:

```text
1 if we finish ahead of the leader
0 otherwise
```

The final decision problem was therefore:

```text
maximize the estimated probability of overtaking the reference leader
```

This does **not** mean the underlying football probabilities changed. It means the optimal action under a winner-take-all objective can differ from the probability vector that minimizes expected match-level log loss.

## The log-loss algebra

Suppose the realized outcome is class `j`.

If our submitted probability is `u_j` and the reference leader's is `d_j`, our gain in total log loss relative to him on that match is

```text
(-log d_j) - (-log u_j) = log(u_j / d_j)
```

If the remaining cumulative deficit is `g`, overtaking him on outcome `j` requires

```text
log(u_j / d_j) > g
```

or equivalently

```text
u_j > exp(g) × d_j
```

This threshold relation is the core of `05_optimize_final_submission_game_theoretic.py`.

## Updating the gap after France–England

Before the third-place match, our submitted row was:

```text
France / Draw / England = 0.46 / 0.30 / 0.24
```

England won in regulation. The public leaderboard did not update before the final submission deadline, so the leader's new score and his France–England row were unknown.

The strategic code therefore does not treat the post-match gap as observed. It reconstructs it from:

1. the last visible total deficit after 30 matches;
2. our known 24% probability on the realized England outcome;
3. an explicit model for the leader's pre-match England probability;
4. uncertainty caused by the leaderboard displaying means rounded to three decimals.

A crucial anti-leakage detail is that the France–England model row must come from a **pre-match** run. A row regenerated after the result is known is not a valid representation of the information available at submission time.

## Modeling the reference leader's final prediction

The working hypothesis was that the reference leader would use a probability shape similar to our unbiased TabPFN output, but sharpened toward Spain.

The unbiased final forecast generated by the pipeline was:

```text
Spain / Draw / Argentina
0.376990 / 0.315075 / 0.307935
```

Candidate leader rows are formed by temperature sharpening this same vector. For temperature `T < 1`:

```text
d_j(T) = p_j^(1/T) / sum_k p_k^(1/T)
```

Because Spain is the largest original cell, sharpening increases Spain’s probability and reduces the draw and Argentina cells. The strategic value of draw and Argentina is therefore not only that the model assigns them substantial combined probability, but also that they are the outcomes on which a Spain-sharpened leader is expected to be relatively vulnerable.

## The final action

The strategic script keeps the two objects separate:

```text
Unbiased probability forecast:
0.376990 / 0.315075 / 0.307935

Competition-aware action:
0.040000 / 0.500000 / 0.460000
```

The second row is **not presented as a superior estimate of the football match**. It is a transparent decision-theoretic transformation designed to create enough relative log-loss gain to overtake the leader under draw or Argentina outcomes, while accepting that a Spain win would almost certainly lose the head-to-head gamble.

All assumptions are exposed in the script: leaderboard checkpoint, number of scored matches, our third-place submission, leader-sharpness temperatures, probability floor, rounding grid, and robustness margins.

# Reproducibility

## Run order at a glance

```powershell
python 01_dixon_coles_ratings.py
python 02_dixon_coles_rating_history.py
python 02b_build_time_aware_style_keywords.py
python 03_predict_finals.py --refresh-results
python 04_anchor_gated_calibration_finals.py
python 05_optimize_final_submission_game_theoretic.py
```

The strategic stage additionally requires a preserved pre-third-place prediction snapshot, as described below.

## 1. Environment

Python 3.10+ is recommended.

```powershell
python -m venv .venv312
Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\.venv312\Scripts\Activate.ps1
python -m pip install --upgrade pip
python -m pip install -r requirements.txt
```

Authenticate the Prior Labs client following the current `tabpfn_client` documentation.

## 2. Input data

The match history is downloaded from the public `martj42/international_results` source by the prediction script when `--refresh-results` is used.

Local data are expected under:

```text
data/raw/
data/interim/
data/output/
data/output_untuned/
```

These directories are ignored by Git. The style-era CSV is a user-maintained input and should be supplied with `--style-eras-csv` when it is not at the default path.

## 3. Build Dixon–Coles ratings

```powershell
python 01_dixon_coles_ratings.py
python 02_dixon_coles_rating_history.py
```

## 4. Build the time-aware style augmentation

```powershell
python 02b_build_time_aware_style_keywords.py
```

This reads:

```text
data/raw/results.csv
```

and generates:

```text
data/raw/results_with_style_keywords_time_aware.csv
data/interim/team_style_keyword_eras_round32_time_aware.csv
```

These outputs are regenerated locally and ignored by Git.

## 5. Run the main final-round model

```powershell
python 03_predict_finals.py --refresh-results
```

For a feature-only audit:

```powershell
python 03_predict_finals.py --refresh-results --features-only
```

For the style-keyword ablation:

```powershell
python 03_predict_finals.py --refresh-results --disable-style-keywords
```

The main output used by the strategic layer is generated locally at:

```text
data/output/final_round_submission_rolling_residual_v3.csv
```

## 6. Run structural calibration

```powershell
python 04_anchor_gated_calibration_finals.py
```

This produces local diagnostics and calibrated probabilities under `data/output/`. They are intentionally not committed.

## 7. Preserve the pre-third-place snapshot

Before updating the match history with the France–England result, preserve the model output used at that time:

```powershell
New-Item -ItemType Directory -Force .\data\output_untuned
Copy-Item `
  .\data\output\final_round_submission_rolling_residual_v3.csv `
  .\data\output_untuned\final_round_submission_rolling_residual_v3.csv
```

This snapshot is required only for reconstructing the information state before the third-place match. It is a generated artifact and remains outside Git.

## 8. Re-run after the third-place result

Refresh the public results and regenerate the final model row:

```powershell
python 03_predict_finals.py --refresh-results
```

## 9. Generate the strategic final action

```powershell
python 05_optimize_final_submission_game_theoretic.py
```

By default the script reads:

```text
data/output_untuned/final_round_submission_rolling_residual_v3.csv
data/output/final_round_submission_rolling_residual_v3.csv
```

and writes a local submission CSV. The file is ignored by Git.

The console report prints:

- the unbiased final model row;
- the reconstructed remaining total-log-loss gap;
- the relative-probability multiplier required to overtake the reference leader;
- the covered draw and Argentina thresholds;
- the final strategic ticket.

# Reproducibility principles

The pipeline follows four rules:

1. **Chronology:** ratings, residuals, and calibration features are constructed only from information available before each match.
2. **Separation of belief and action:** the raw TabPFN probability vector is retained even when the competition action differs.
3. **Explicit assumptions:** opponent behavior and leaderboard uncertainty are configuration values, not hidden manual edits.
4. **No committed outputs:** the repository contains the code required to regenerate results, not generated predictions or submission files.

This separation makes it possible to evaluate the forecasting model independently from the final game-theoretic decision.
